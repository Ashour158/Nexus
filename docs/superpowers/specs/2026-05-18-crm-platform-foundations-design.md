# CRM Platform Foundations Design

## Purpose

This spec defines the system-wide CRM foundations needed before the next feature build: admin-controlled record coding, document templates with DOCX/PDF export, universal import/export, enforced contact-account linking, and a deeper account data model. These are platform contracts, not page-level enhancements, so they must be implemented through shared schemas, service APIs, admin UI, and preview data together.

## Scope

In scope:

- Reference/coding rules for all first-class CRM objects.
- Document templates for quotes first, with an extensible model for contracts, proposals, account letters, and service documents.
- Import/export registry for all modules.
- Required account linkage for contacts.
- Expanded account profile, billing address, and shipping address fields.
- Admin screens for coding rules, document templates, import/export settings, and account/contact governance.
- Local preview API support so the UI remains usable without every backend service running.

Out of scope:

- SaaS billing.
- AI scoring, AI template writing, or AI document generation.
- Replacing the existing microservice architecture.
- Full workflow automation execution beyond emitting events and validation hooks needed by these foundations.

## Existing System Fit

The repo already has useful pieces:

- `services/data-service` owns import/export jobs, saved views, audit logs, and recycle bin.
- `services/metadata-service` owns custom fields, validation rules, field permissions, tags, and duplicate groups.
- `services/accounts-service` and `services/contacts-service` both contain account/contact models, but contact `accountId` is currently optional.
- `services/quotes-service` has `Quote.quoteNumber`, but not a generalized coding engine or template render pipeline.
- `apps/web/src/app/(dashboard)/settings` already contains settings sections that can host the admin screens.
- `apps/web/src/lib/server/dev-preview-data.ts` now acts as the local preview state for page wiring.

The design keeps these services and adds clear ownership instead of creating an all-purpose monolith.

## Architecture

### 1. Coding Engine

Create a tenant-scoped coding system controlled by administrators.

Record code examples:

- Account: `ACC-2026-000001`
- Contact: `CON-2026-000001`
- Lead: `LED-2026-000001`
- Product: `PRD-2026-000001`
- Quote: `QTE-2026-000001`
- Deal: `OPP-2026-000001`
- Activity: `ACT-2026-000001`
- Document: `DOC-2026-000001`

Core data:

- `entityType`: `account`, `contact`, `lead`, `deal`, `product`, `quote`, `activity`, `document`, plus future modules.
- `prefix`: admin-defined string such as `ACC`.
- `pattern`: tokenized format such as `{PREFIX}-{YYYY}-{SEQ:6}`.
- `sequenceScope`: `tenant`, `year`, `month`, or `entity`.
- `nextSequence`: stored per entity/scope.
- `isManualOverrideAllowed`: whether admins can override generated codes.
- `isRequired`: whether the entity must have a code before save.
- `lockedAfterCreate`: whether code changes are prohibited after creation.
- `previewCode`: generated by API for admin UI previews.

Behavior:

- All create endpoints for coded modules call the coding engine before persistence.
- Codes are unique per `tenantId + entityType + code`.
- Code allocation is atomic and must not skip or duplicate under concurrent creates.
- Admin changes only affect future generated codes.
- Any manual override is permission-gated and audited.
- Record detail pages display code beside the entity name.
- List tables include code as a default visible column.

Recommended ownership:

- Store rules in `metadata-service`, because it already owns tenant metadata, validation rules, custom fields, tags, and duplicate governance.
- Expose internal allocation endpoint for services: `POST /internal/codes/:entityType/allocate`.
- Expose admin endpoints: `GET/POST/PATCH /api/v1/coding-rules`.

### 2. Document Template Engine

Create a deterministic document template system. Quote templates are the first production use case.

Template data:

- `id`, `tenantId`, `module`, `name`, `description`.
- `status`: `DRAFT`, `ACTIVE`, `ARCHIVED`.
- `version`.
- `sourceFormat`: `DOCX`, `HTML`, or `MARKDOWN`.
- `templateBody` or `fileId`.
- `mergeFields`: approved placeholders such as `{{quote.quoteNumber}}`, `{{account.name}}`, `{{contact.fullName}}`.
- `lineItemBlocks`: repeatable rows for quote items.
- `header/footer` configuration.
- `locale`, `currency`, `pageSize`, `orientation`.
- `createdBy`, `updatedBy`, timestamps.

Rendering behavior:

- Users select a template from quote detail or quote editor.
- The system merges quote, account, contact, product, tax, totals, payment terms, and signature data.
- Users can preview before export.
- Export formats: `docx` and `pdf`.
- Every rendered document stores a versioned output record linked to quote/account/contact/deal.
- Exports are permission checked and audit logged.

Recommended ownership:

- Use `document-service` for template storage and rendering APIs.
- Use `quotes-service` to provide canonical quote data snapshots.
- Store generated files through `storage-service`.
- Add web API facade routes under `apps/web/src/app/api/templates` and quote-specific export actions.

### 3. Universal Import/Export

Standardize import/export through a registry of supported modules.

Supported modules:

- Accounts
- Contacts
- Leads
- Deals
- Products
- Quotes
- Activities
- Documents metadata
- Notes
- Users where admin permissions allow

Import behavior:

- Upload CSV or XLSX.
- Detect columns.
- Map source fields to target fields.
- Preview first rows.
- Validate required fields and data types before execution.
- Enforce account requirement for contacts.
- Offer duplicate strategy: skip, update existing, create duplicate, or merge candidate.
- Produce an import job with counts, failed rows, and downloadable error file.

Export behavior:

- Export filtered list views.
- Export selected rows.
- Export visible columns or all columns.
- CSV and XLSX for data modules.
- PDF/DOCX only through document templates.
- Respect RBAC, field permissions, and tenant boundaries.

Recommended ownership:

- Extend `data-service` import/export routes and services.
- Add a module registry in shared code so web, data-service, and validation agree on fields.
- Local preview should support import/export for at least accounts, contacts, leads, products, and quotes.

### 4. Contact Must Belong To Account

Contacts are not standalone records. Every contact must be linked to exactly one account at creation time.

Rules:

- `CreateContactSchema.accountId` becomes required.
- Contacts UI must require account selection before save.
- Lead conversion must either choose an existing account or create a new account before contact creation.
- Imports must reject contact rows without `accountId`, `accountCode`, or enough account fields to create/match an account.
- API updates may move a contact to another account only with permission.
- Contact detail pages show the linked account as primary context.
- Account detail pages show contacts as a first-class related list.

Migration behavior:

- Existing contacts with no account must be moved to a generated holding account per tenant, such as `Unassigned Contacts`.
- After migration, database constraints should prevent null `accountId`.

### 5. Deep Account Model

Accounts become the commercial anchor for the CRM.

Core fields:

- `code`, `name`, `legalName`, `tradeName`.
- `ownerId`, `parentAccountId`, `territoryId`.
- `website`, `email`, `phone`, `fax`, `linkedinUrl`.
- `type`, `tier`, `status`, `lifecycleStage`.
- `industry`, `subIndustry`, `sicCode`, `naicsCode`.
- `annualRevenue`, `employeeCount`, `foundedYear`.
- `taxId`, `vatNumber`, `commercialRegistrationNumber`.
- `paymentTerms`, `creditLimit`, `currency`, `priceBookId`.
- `healthScore`, `npsScore`, `riskLevel`, `lastActivityAt`.
- `description`, `tags`, `customFields`, `dataQualityScore`.

Billing address:

- `billingAddressLine1`
- `billingAddressLine2`
- `billingCity`
- `billingState`
- `billingPostalCode`
- `billingCountry`
- `billingLatitude`
- `billingLongitude`

Shipping address:

- `shippingAddressLine1`
- `shippingAddressLine2`
- `shippingCity`
- `shippingState`
- `shippingPostalCode`
- `shippingCountry`
- `shippingLatitude`
- `shippingLongitude`
- `shippingInstructions`
- `sameAsBilling`

Related data:

- Contacts
- Deals
- Quotes
- Documents
- Activities
- Notes
- Account hierarchy
- Consents and preferences
- Audit history
- Import/export history

UI behavior:

- Account list shows code, name, owner, type, tier, status, country, open deals, contacts, and last activity.
- Account detail uses sections: Overview, Contacts, Deals, Quotes, Documents, Activities, Addresses, Commercial, Compliance, History.
- Account create/edit form separates billing and shipping address and supports copying billing to shipping.

New permissions:

- `coding-rules:read`
- `coding-rules:update`

- `data-export:run`

Admin-only defaults:

- Coding rule changes.
- Manual code override.
- Template activation/archive.
- Global import/export field mappings.
- Migrating unlinked contacts.

## Events And Integration

Emit events for downstream services:

- `coding.rule.updated`
- `record.code.assigned`
- `document.template.created`
- `document.template.activated`
- `document.rendered`
- `data.import.started`
- `data.import.completed`
- `data.export.completed`
- `contact.account_linked`
- `account.updated`

Events must include `tenantId`, `actorId`, `correlationId`, and changed entity ids.

## Error Handling

- Duplicate generated code returns `409 CODE_CONFLICT` and retries allocation once internally.
- Invalid code pattern returns `400 INVALID_CODE_PATTERN`.
- Contact create without account returns `400 CONTACT_ACCOUNT_REQUIRED`.
- Template render with missing required merge field returns `422 TEMPLATE_MERGE_FAILED`.
- Import validation errors stay in the import job and do not partially create invalid rows.
- Export requests exceeding row limits return `413 EXPORT_TOO_LARGE` with a suggestion to narrow filters.

## Testing

Minimum coverage:

- Unit tests for code pattern parsing and sequence allocation.
- Concurrent allocation test for same tenant/entity.
- Validation tests proving contacts require accounts.
- Account schema tests for billing/shipping fields.
- Template render tests for quote merge fields and line-item tables.
- Import preview and validation tests for accounts/contacts.
- Export tests for filtered views and selected columns.
- Web tests for admin coding UI, account form, contact form, quote template export action.

## Implementation Order

1. Shared schemas and types for codes, addresses, document templates, and import/export module registry.
2. Metadata-service coding rule storage and allocation API.
3. Account and contact schema migrations.
4. Validation updates: contacts require account; account address fields validate consistently.
5. Service create/update flows call coding engine and enforce account linkage.
6. Data-service module registry and import/export validation.
7. Document-service template storage and quote render/export flow.
8. Web settings pages for coding rules, templates, and import/export.
9. Account and contact UI updates.
10. Local preview wiring and smoke tests.

## Acceptance Criteria

- Admin can configure and preview reference codes per module.
- New accounts, contacts, leads, products, quotes, deals, activities, and documents receive codes.
- Codes are visible in lists and detail pages.
- Contacts cannot be created without an account.
- Lead conversion always links the resulting contact to an account.
- Accounts contain separate billing and shipping addresses.
- Quote templates can be created, selected, rendered, and exported as DOCX/PDF.
- All primary modules have import/export entry points.
- Imports validate required fields and produce clear row-level errors.
- Exports respect filters, columns, permissions, and tenant isolation.
- Local preview remains usable on `localhost:3000` without requiring every backend service.

## Open Decisions

Recommended defaults unless changed during implementation:

- Coding rules live in `metadata-service`.
- Template rendering lives in `document-service`.
- Quote data snapshot comes from `quotes-service`.
- Generated files are stored via `storage-service`.
- Contact migration uses one tenant-level holding account named `Unassigned Contacts`.
- First export formats are CSV/XLSX for data and DOCX/PDF for templates.
