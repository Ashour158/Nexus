# Strict CPQ, RFQ, Quote, DRQ Workflow Design

## Purpose

Rebuild CPQ around a strict enterprise workflow: sellers create RFQs from commercial context, the system validates and prices products, approvals govern quote generation and discounts, and only approved quote packages can be sent, downloaded, or signed. Normal users must not choose quote templates/packages or expose governance internals on the quote page. Admins control exceptions through low-code validation, workflow, template, and policy settings.

## Core Business Flow

1. **Start from commercial context**
   - RFQ must be created from an existing deal.
   - The deal must be linked to an account and may be linked to a contact or lead.
   - No orphan RFQs, quotes, DRQs, or orders are allowed.

2. **Build RFQ with products**
   - RFQ captures requested products, quantities, configuration, required-by date, delivery location, currency, tax zone, attachments, and customer requirements.
   - Product lines are normalized: `productId`, `sku`, `name`, `quantity`, `uom`, `unitPrice`, `discountPercent`, `taxPercent`, `lineTotal`, `margin`, `availability`, and validation status.
   - Product rules validate required configuration, active price book, territory/currency availability, stock/service eligibility, incompatible bundles, minimum quantity, and floor margin.

3. **Review quote draft**
   - Converting RFQ creates a quote draft through CPQ service logic only.
   - Quote draft inherits RFQ lines, account/deal/contact/lead context, price book, tax zone, payment terms, validity period, owner, and approver route.
   - Seller sees commercial summary, line items, validation issues, and next action.
   - Seller does not choose quote package/template.

4. **Approval workflow**
   - Quote approval path is generated by workflow policy.
   - Approval map appears on RFQ and quote pages: current stage, current approver, completed approvers, timestamps, SLA timers, rejected/returned reason, and next action.
   - Approval stages can include Sales Manager, Finance, Legal, Product, Executive, or custom admin-created stages.
   - Every action writes an audit event, outbox event, actor, timestamp, status transition, and immutable snapshot.

5. **Approved quote package**
   - After final approval, the system renders the approved package automatically from the active admin-governed template.
   - Users may download PDF, send by email/portal, or initiate e-sign only after approval and package generation.
   - DOCX and HTML rendering are admin/governance artifacts unless admin explicitly exposes them to a role.

6. **Customer signature and acceptance**
   - E-sign tracks envelope status: `NOT_SENT`, `SENT`, `VIEWED`, `SIGNED`, `DECLINED`, `EXPIRED`.
   - Signed quote can convert to order only if quote is approved, unexpired, not superseded, and has accepted signature or authorized manual acceptance.

7. **DRQ discount workflow**
   - Seller requests discount from the quote page.
   - DRQ requires requested discount percent, prevalidated reason, reason notes, expected winning probability if approved, business impact, and optional competitor/reference evidence.
   - DRQ follows its own approval map with timestamps and SLA.
   - Approved DRQ generates a new immutable quote revision with recalculated totals and a new approved-package requirement.
   - Rejected DRQ preserves the current quote and records reason.

## UI Model

### RFQ Detail Page

Primary user page for commercial build-up.

- Header: RFQ number, deal/account/contact context, status, required-by date, owner.
- Process map: Deal -> RFQ -> Review -> Quote Approval -> Approved Package -> Send/Sign.
- Product grid: editable before submission, read-only after review submission.
- Validation panel: missing data, blocked products, price book issues, margin warnings, tax/territory rules.
- Attachments: customer RFQ files, drawings, scope documents.
- Actions: save draft, submit for review, convert/review quote, view generated quote.

### Quote Detail Page

Seller-facing page, cleaned of admin/governance clutter.

- Header: quote number, status, version, expiry date, linked RFQ/deal/account/contact.
- Approval map: live stage, approvers, timestamps, SLA, action history summary.
- Commercial summary: subtotal, discount, tax, total, payment terms, validity.
- Line items: customer-friendly product and pricing table.
- DRQ panel: request discount when allowed by status and role.
- Customer actions: download approved PDF, send to customer, portal share, send for e-sign, convert to order.
- Hidden from normal users: template selection, package rendering controls, checksum, rendered package internals, governance internals, admin policy controls.

### Admin Settings

Admin controls everything that normal sellers cannot touch.

- Quote templates/packages: active template, role exposure, versioning, DOCX upload validation, PDF rendering policy, locale/currency variants.
- Workflow rules: approval stages, thresholds, SLA timers, return/reject behavior, escalation.
- Validation policies: required fields by module, field conditions, product/rule dependencies, per-role exceptions.
- Low-code extension: admins can add workflow nodes, validations, actions, and allowed bypasses without code changes.

### Journey Navigation

The current sidebar item `Journeys` is not a standalone seller module. Move it under workflow/admin settings or rename it to **Customer Journey Rules** and use it as an embedded journey engine for lifecycle states, milestones, and automation hooks.

## Backend Engines

### CPQ Rule Engine

Responsible for price, product, tax, margin, bundle, territory, and eligibility checks.

Outputs:
- normalized quote/RFQ lines
- validation issues
- pricing waterfall
- margin and discount flags
- approval requirement

### Workflow and Approval Engine

Responsible for quote and DRQ approval routes.

Outputs:
- approval map
- current approver
- allowed actions
- SLA state
- approval timestamps
- audit/outbox events

### Template Rendering Engine

Responsible for governed package generation.

Rules:
- normal users cannot select templates
- active template is resolved by admin policy
- every rendered package links to quote revision
- PDF is the default customer-facing output
- DOCX/HTML are admin/role controlled

### Revision and Immutability Engine

Responsible for snapshots and supersession.

Rules:
- approved/sent/signed quotes cannot be edited in place
- DRQ approval creates a new revision
- each revision stores lines, totals, terms, approval state, template version, actor, timestamp, and reason
- only current active revision can be sent or signed

### Audit, Events, and Timeline Engine

Responsible for nervous-system wiring.

Events:
- `rfq.created`
- `rfq.product_added`
- `rfq.submitted_for_review`
- `quote.generated_from_rfq`
- `quote.validation_failed`
- `quote.submitted_for_approval`
- `quote.approved`
- `quote.rejected`
- `quote.package_rendered`
- `quote.sent`
- `quote.signature_requested`
- `quote.signed`
- `drq.requested`
- `drq.approved`
- `drq.rejected`
- `quote.revised_from_drq`
- `order.created_from_quote`

Events update contact/account/deal timelines where relevant.

## Validation and Permissions

Strict baseline:
- normal users cannot create standalone quotes
- RFQ requires deal, account, owner, currency, required-by date, and at least one valid product line
- quote requires RFQ, deal, account, owner, approver route, template policy, validUntil, and non-empty normalized lines
- DRQ requires quote, discount percent, reason code, reason notes, winning probability, and approver hierarchy
- send/sign/download customer package requires approved current revision and rendered PDF
- order conversion requires approved accepted/signed quote, not expired, not superseded

Admin low-code can configure:
- required fields by module and condition
- quote template selection policy
- approval thresholds by discount, margin, total, territory, role, product family, and customer tier
- allowed bypasses with approval and audit
- SLA timers and escalations
- role visibility for package internals

## Error Handling

- Validation errors return structured field-level errors.
- Workflow violations return allowed next actions.
- Approval errors explain missing approver, SLA breach, or policy mismatch.
- Template/rendering errors remain admin-visible, while sellers see "package generation pending/failed, contact admin."
- Stale revisions cannot be sent; user is redirected to current revision.

## Testing

Add E2E and API coverage for:
- RFQ cannot be submitted without deal/account/product lines.
- RFQ product validation blocks invalid products.
- RFQ converts to quote only through review path.
- Quote cannot be sent before approval/package render.
- Normal user cannot choose template/package.
- Admin can configure template policy.
- DRQ approval creates a new quote revision.
- Rejected DRQ does not change quote totals.
- Approval map timestamps update after each action.
- Signed quote converts to order.

## Implementation Approach

Recommended approach: **strict workflow-first**.

Build in phases:
1. Hide/move quote package and governance internals from seller quote page.
2. Add shared CPQ workflow map component to RFQ, quote, and DRQ views.
3. Harden RFQ line item creation, validation, and conversion to quote.
4. Harden quote lifecycle transitions and approval-gated customer actions.
5. Harden DRQ approval and revision generation.
6. Move template/package controls to admin settings.
7. Wire audit/outbox/timeline events across account/contact/deal.
8. Add E2E workflow tests.
